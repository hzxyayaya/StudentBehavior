# Rewrite Scope

Last Updated: 2026-03-20

## 目标

本项目下一阶段不是继续修补旧实现，而是:

- 先提取旧项目中真正有价值的业务能力与约束
- 再删除旧实现代码
- 最后基于保留能力，重做一份更干净的新版

因此，下面的清单区分的是“需要继承的能力”与“可以丢弃的旧实现形式”，不是对旧代码质量的简单评价。

## 必须保留的核心能力

### 1. 学生风险识别主线

新版应继续保留以下业务主线:

- 基于学生行为与学业数据识别风险学生
- 支持全校层面的风险概览
- 支持学生个体层面的画像与解释

保留原因:

- 这是前端页面、后端接口、模型服务和数据集共同指向的主目标
- 也是当前项目最稳定的业务中心

### 2. 核心业务页面

新版建议保留以下页面能力，但允许完全重做页面结构与交互:

- 登录入口
- 仪表盘/概览页
- 风险学生列表页
- 学生画像详情页
- 系统设置或模型管理入口

保留原因:

- 这些页面与当前路由、组件结构、API 划分高度一致
- 它们覆盖了“总览 -> 筛选 -> 下钻 -> 分析/管理”的主工作流

### 3. 核心后端能力域

新版建议保留以下后端能力域:

- 认证
- 学生信息与画像查询
- 风险预警列表
- 全校分析统计
- 报告生成与解释
- 模型训练/预测
- 特征计算

保留原因:

- 这些能力已经在 `app/main.py` 的路由注册中形成稳定边界
- 它们可以直接转化为新版的服务模块和接口分组

### 4. 数据驱动架构

新版应继续保留“从多源学生数据生成行为特征，再做分析与解释”的总体思路。

必须保留的层次:

- 原始数据源层
- 特征工程层
- 风险预测/聚类层
- 可解释与报告层

保留原因:

- 当前数据目录和模型服务都围绕这条链路组织
- 如果去掉这条链路，新版就不再是同一个项目

### 5. 可解释输出能力

新版建议保留以下解释型输出:

- 学生画像摘要
- 风险因素解释
- 可供教师/管理者阅读的分析报告

说明:

- “必须保留”的是解释能力本身
- 不一定要原样保留 SHAP 图、原文案格式或当前报告组织方式

## 建议保留但允许降级的能力

这些能力需要保留，其中模型训练/聚类管理已明确进入 V1:

- 聚类分析的完整展示链路
- 模型训练与聚类管理能力
- 管理员专属设置页
- 复杂图表的丰富视觉表现
- 一次性接入所有现有 Excel 数据源

处理建议:

- V1 保留最必要版本
- 先打通数据 -> 风险 -> 画像 -> 解释主链路，同时建设一个相对完整的训练/聚类管理模块
- 训练参数、评估指标、结果查看和管理流程需要纳入正式范围文档

## 可以废弃的旧实现

### 1. 旧前端的具体组件实现

可以废弃:

- 现有组件拆分方式
- 现有 CSS 体系
- 现有页面布局和视觉风格
- 现有 store 组织方式
- 依赖 mock 的页面状态流

原因:

- 这些属于实现细节，不是业务能力本身
- 你要的是更干净的新版，不应该被旧 UI 结构绑死

### 2. 旧后端的具体代码结构

可以废弃:

- 当前模块命名和目录组织
- 当前 ORM/schema/service 的拆分方式
- 当前路由文件内部实现
- 当前模型持久化文件的管理方式

原因:

- 这些是技术实现，不是必须继承的产品能力
- 新版应以更清晰的边界和更易维护的结构重建

### 3. 过重或不稳定的扩展能力

可以延后甚至暂时废弃:

- 不是主链路所必需的复杂图表
- 首版不关键的高级管理功能
- 暂时无法稳定复现的训练/评估细节
- 未验证可运行的旧环境耦合逻辑

原因:

- 新版应优先保留“核心闭环”，而不是一次搬运全部复杂度

## 删除旧代码前必须完成的提取

在真正删除现有实现前，至少要补齐这些文档:

1. 保留能力清单
2. 可废弃实现清单
3. V1 重写目标
4. 关键数据输入与输出定义
5. 最小页面/接口清单

## 当前建议的新版 V1 范围

如果按“只保留核心能力，做更干净的新版”推进，V1 推荐只保留:

- 登录
- 风险总览仪表盘
- 风险学生列表
- 学生画像详情
- 基础风险解释/报告
- 最小可用的数据读取与特征计算链路
- 相对完整、可配置、可查看结果的模型训练与聚类管理模块

V1 可以暂缓:

- 全量数据源接入
- 复杂可视化和高级设置
- 非主链路实验性功能

## 模型训练/聚类管理的当前共识

新版 V1 中，这一模块不应只是一个“触发按钮”或简单入口，而应至少覆盖:

- 可配置的训练参数输入
- 聚类相关参数配置
- 训练任务触发与状态反馈
- 训练/聚类结果查看
- 关键指标或结果摘要展示
- 面向管理者的基础操作界面

当前仍待细化的部分:

- 具体参数项清单
- 是否需要保存历史训练记录
- 是否需要对比不同训练结果
- 是否需要权限分层和审计信息

## 当前结论

旧项目里真正值得继承的不是“现有代码写法”，而是:

- 学生行为风险预警这个业务目标
- 数据到画像再到解释的核心闭环
- 支撑该闭环的最小页面与接口集合

旧代码本身大部分可以作为参考样本，而不是新版的直接基础。
