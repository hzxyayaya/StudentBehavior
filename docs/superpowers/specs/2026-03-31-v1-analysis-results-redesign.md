# V1 Analysis Results Redesign

Status: Approved Design Baseline

Last Updated: 2026-03-31

## 1. 文档目的

本文档用于替换当前 V1 中过度依赖“四象限”表达的业务口径，重新冻结本项目的上层目标、系统输出、接口表达与页面叙事。

本次重构的目标不是推翻现有数据链路，而是：

- 保留现有多源数据接入与学期级特征构建能力
- 将上层业务目标统一为 4 类分析任务
- 将系统交付物统一为 10 个分析结果
- 将“群体分层”从固定四象限表达调整为更通用的行为模式识别
- 为后续文档、API、结果层和前端改造提供单一事实来源

## 2. 重构背景

当前仓库中的主链路已经形成：

1. `projects/semester-etl`
2. `projects/analytics-db`
3. `projects/model-stubs`
4. `projects/demo-api`
5. `projects/demo-web`（独立工作树）

这条链路已经能支撑“学业风险动态感知与预警”的 V1 Demo，但上层业务表达仍然存在两个问题：

1. 当前系统对“四象限”表达绑定过重，容易把群体分析收窄为单一视觉框架，而不是面向题目本身的开放式行为模式识别
2. 当前文档与接口更偏向“风险预警 Demo”，尚未完整映射到赛题要求中的 4 类分析任务与 8~10 个分析结果

因此，本次改造要做的是“口径收口”，不是“底层推倒重来”。

## 3. 新的上层目标

项目统一围绕以下 4 类分析任务组织：

1. 学业风险动态感知与预警
2. 学业轨迹演化与关键行为分析
3. 学生个体画像与群体分层分析
4. 发展方向与去向关联分析

这 4 类任务共同构成系统对外叙事，不再将“四象限群体模型”视为核心目标本身。

## 4. 系统输出重定义

V1 统一输出以下 10 个分析结果：

1. 学生个体画像
2. 学生群体画像
3. 行为模式识别
4. 学业风险概率分层
5. 风险等级预警
6. 关键影响因子解释
7. 个性化干预建议
8. 学期趋势分析
9. 专业/学院群体对比
10. 模型摘要说明

这 10 个结果是系统对外的固定交付物。任何页面、接口、报告和答辩表达都必须能映射到这 10 个结果之一。

## 5. 与既有方案的关系

### 5.1 保留的内容

本次重构保留以下已有基础：

- 多源行为数据整合
- 学号统一与学期粒度聚合
- 学期级行为特征宽表
- 基于综合测评低等级风险的预警主线
- 风险概率、风险等级、解释、建议、趋势和专业对比等结果层结构
- 当前前后端 Demo 的主展示链路

### 5.2 替换的内容

本次重构替换以下旧口径：

- 不再把“四象限”作为系统的唯一群体分析表达
- 不再把 `quadrant_label` 作为业务主字段
- 不再将“群体页 = 四象限页”视为固定页面形态
- 不再用“四象限”解释所有群体、风险和报告结果

### 5.3 弱化而非立即删除的内容

现有代码中的四象限逻辑在短期内可以保留为历史兼容实现，但必须满足：

- 不再作为对外主叙事
- 不再作为新文档中的冻结表达
- 后续结果层应逐步迁移为 `behavior_pattern` / `group_segment`

## 6. 新的业务语义

### 6.1 行为模式识别

“行为模式识别”表示系统基于学期级行为特征，对学生群体做结构化分层与模式总结。

它的目标是：

- 识别典型行为模式
- 解释不同群体的特征差异
- 支撑后续风险预警和个体下钻

它不再预设必须是“四象限”，也不强制固定为 4 个标签。实际落地时可以是：

- 规则分层
- 聚类分层
- 基于特征阈值的管理型标签

业务输出应优先使用可理解的人群标签，例如：

- 学习投入稳定组
- 作息失衡风险组
- 课堂参与薄弱组
- 综合发展优势组

### 6.2 风险主线

风险主线保持不变：

- 核心监督标签仍为“综合测评低等级风险”
- 仍按学期粒度进行风险预测与趋势展示
- 风险输出仍包含 `risk_probability` 和 `risk_level`

### 6.3 解释与建议

解释层和建议层仍然是系统必要组成，但未来应逐步从规则模板过渡到：

- 数据驱动的关键因子解释
- 规则骨架 + LLM 生成的个性化建议

当前如果仍使用 stub 或规则版结果，必须在文档和页面上明确标注边界。

## 7. 技术链路调整

重构后的技术链路定义为：

1. 多源数据接入与清洗
2. 学号统一与学期级聚合
3. 学生行为特征宽表构建
4. 行为模式识别与群体分层
5. 基于综合测评低等级标签进行风险预测
6. 输出风险概率分层与风险等级预警
7. 生成关键影响因子解释
8. 生成个性化干预建议与报告
9. 输出趋势、专业/学院对比与模型摘要
10. 通过管理视角页面完成展示与下钻

其中第 4 步不再限定为“四象限映射”。

## 8. 页面叙事重构

### 8.1 页面闭环

V1 页面闭环保持：

1. 登录页
2. 总览页
3. 群体分析页
4. 风险预警页
5. 学生个体页

### 8.2 页面职责调整

总览页：

- 展示风险分层
- 展示趋势变化
- 展示专业/学院对比
- 展示重点群体分层摘要
- 展示模型摘要

群体分析页：

- 展示学生群体画像
- 展示行为模式识别结果
- 展示群体特征差异
- 支撑从群体下钻到预警列表

风险预警页：

- 展示风险等级预警
- 支持 `term`、`risk_level`、`major_name`、`group_segment` 筛选
- 支撑从预警列表下钻到个体页

学生个体页：

- 展示学生个体画像
- 展示风险概率与等级
- 展示关键影响因子解释
- 展示干预建议
- 展示学期趋势
- 展示发展方向与去向关联摘要

## 9. API 重构要求

现有 API 的重构要求如下：

### 9.1 保留的接口

- `POST /api/auth/demo-login`
- `GET /api/analytics/overview`
- `GET /api/warnings`
- `GET /api/students/{student_id}/profile`
- `GET /api/students/{student_id}/report`
- `GET /api/models/summary`

### 9.2 替换或新增的接口

应逐步用以下接口替换或补强旧的四象限接口：

- `GET /api/analytics/groups`
- `GET /api/analytics/trends`
- `GET /api/analytics/majors`
- `GET /api/analytics/directions`

### 9.3 字段迁移方向

应从以下旧字段迁移：

- `quadrant_label`

迁移到以下更通用字段之一：

- `behavior_pattern`
- `group_segment`

具体命名可在实现阶段二选一，但必须全链路统一。

## 10. 结果层重构要求

`projects/model-stubs` 后续应满足：

- 不再把 `quadrant_label` 作为核心输出字段
- 将群体标签改为 `behavior_pattern` 或 `group_segment`
- 保留并补强以下结果字段：
  - `risk_probability`
  - `risk_level`
  - `top_factors`
  - `intervention_advice`
  - `term_trend`
  - `major_name`
  - `college_name`
  - `development_direction`（如可从现有数据稳定派生）

## 11. 前端重构要求

`projects/demo-web` 后续应满足：

- 删除“四象限”作为主视觉和主业务表达
- 将群体页替换为“行为模式分层”或“重点群体分析”
- 将预警页中的 `quadrant_label` 筛选替换为 `group_segment`
- 将学生页中的“四象限标签”替换为“行为模式标签”
- 页面文案统一映射到 4 类分析任务与 10 个分析结果

## 12. 验收标准

本次重构完成的判断标准是：

1. 上层文档不再以“四象限”作为冻结业务语义
2. API contract 不再把 `quadrant_label` 作为共享枚举核心字段
3. 页面和接口能完整映射到 4 类分析任务
4. 系统输出能完整映射到 10 个分析结果
5. 现有 Demo 主链路仍可运行
6. 若仍存在 stub 字段，边界表述清晰

## 13. 非目标

本次重构不在本轮直接完成以下事项：

- 正式模型训练替换
- AUC 指标补齐
- 大模型报告生成链路上线
- 全量去除所有历史四象限代码

本轮优先目标是统一叙事、统一接口表达、统一结果结构。
