# 风险标签映射与模型设计说明

Last Updated: 2026-03-25

## 1. 文档目的

本文档用于定义 V1 阶段的风险标签来源、特征输入、群体划分、预测建模与解释生成逻辑，作为后续数据处理、模型实现、接口设计和答辩说明的统一依据。

本文档重点回答以下问题：

- 风险标签从哪里来，如何映射
- 模型使用哪些输入特征
- 四象限群体如何从聚类结果映射出来
- 风险预测、解释和干预建议之间如何衔接

## 2. 模型设计总览

V1 阶段的模型链路建议固定为以下结构：

1. 多源数据清洗与学号统一
2. 按学期构建学生特征宽表
3. 基于行为特征进行群体聚类
4. 将聚类结果映射为四象限群体标签
5. 基于综合测评低等级标签训练风险预测模型
6. 基于特征贡献结果生成归因解释
7. 基于规则模板和 LLM 润色生成干预建议

这条链路的设计目标不是追求复杂模型堆叠，而是保证每一层都能被解释、展示和复用。

## 3. 风险标签设计

## 3.1 标签来源

V1 的核心风险标签来源于 `本科生综合测评.xlsx` 中的等级字段。

该等级字段对应学生在某学期或某评价周期内的综合表现结果，是当前最适合作为监督标签的业务指标。

## 3.2 标签定义原则

V1 建议优先采用二分类风险标签：

- 正类：综合测评低等级学生
- 负类：非综合测评低等级学生

采用二分类的原因：

- 更适合当前比赛指标中的 AUC 评估
- 更容易构建清晰的风险预警逻辑
- 更方便在前端展示高、中、低风险分层结果

## 3.3 待细化项

在正式建模前，需要进一步确认以下细节：

- 综合测评等级字段的具体取值有哪些
- 哪些等级应归入“低等级”
- 是否需要先做等级标准化或缺失值清洗
- 标签按学期、学年还是评价周期与行为数据对齐

当前建议：

- 保持“按学期对齐”的整体原则
- 若综合测评原始周期与学期不完全一致，可通过最近周期映射或规则对齐处理

## 4. 特征输入设计

## 4.1 特征输入原则

V1 的特征不直接使用全部原始字段，而是先在学期粒度上聚合成具有业务含义的行为指标。

这样做的原因：

- 降低原始数据噪声
- 使特征更适合解释与展示
- 便于后续在画像页和报告中复用

## 4.2 八大特征维度

V1 建议统一围绕以下 8 个维度构建特征：

1. 学业基础表现
2. 课堂学习投入
3. 在线学习积极性
4. 图书馆沉浸度
5. 网络作息自律指数
6. 早晚生活作息规律
7. 体质及运动状况
8. 综合荣誉与异动预警

## 4.3 典型特征示例

建议每个维度至少沉淀 2 到 4 个可计算指标，形成一套稳定特征表。

示例包括：

- GPA、挂科次数、低分课程数
- 出勤率、迟到次数、缺勤次数
- 作业完成率、考试参与率、在线学习完成率
- 图书馆打卡频次、平均停留时长
- 深夜上网天数、总上网时长
- 晚归次数、作息波动程度
- 跑步打卡频率、体测成绩
- 奖学金情况、竞赛情况、学籍异动信号

## 5. 群体聚类与四象限映射

## 5.1 聚类目标

聚类模块的目标不是直接给学生打“好坏标签”，而是从群体层面发现行为模式差异。

V1 中聚类结果最终需要服务于四象限叙事：

- 自律共鸣型
- 被动守纪型
- 脱节离散型
- 情绪驱动型

## 5.2 聚类输入

建议使用经过标准化处理后的学期级行为特征作为聚类输入。

尤其应优先覆盖与“投入度”和“稳定性”强相关的特征，例如：

- 出勤与课堂参与
- 作业与考试完成
- 在线学习参与
- 图书馆与学习活动投入
- 深夜上网与作息规律
- 学期内波动性指标

## 5.3 聚类方法

V1 可优先采用以下路线：

- 基础方案：KMeans
- 增强方案：在对比实验中补充层次聚类或 GMM，但不作为首要实现路径

选择 KMeans 的原因：

- 实现简单
- 易于解释聚类中心
- 容易和四类模式做固定映射

## 5.4 四象限映射逻辑

聚类结果产生后，不直接把数值类别暴露给用户，而是通过解释层映射为四象限标签。

映射逻辑建议如下：

1. 基于聚类中心分析各群体在“投入度”相关特征上的整体高低
2. 基于聚类中心分析各群体在“稳定性”相关特征上的整体高低
3. 将每个聚类群体映射到四象限中的一个标签
4. 输出固定的业务名称与解释文本

## 5.5 四象限辅助指标

为了让映射更稳定，建议额外构建两个复合指标：

- 投入度指数
- 稳定性指数

这两个指数不一定直接参与模型训练，但可以作为：

- 四象限可视化坐标
- 聚类解释辅助依据
- 个体画像展示项

## 6. 风险预测模型设计

## 6.1 目标

风险预测模块用于预测学生落入综合测评低等级的概率。

## 6.2 输入

- 学期级特征宽表
- 清洗后的风险标签
- 必要的训练、验证、测试集切分

## 6.3 输出

- 每名学生的风险概率
- 风险等级分层结果
- 模型指标，例如 AUC

## 6.4 候选算法

V1 建议优先使用树模型路线：

- LightGBM
- XGBoost

优先原因：

- 对结构化表格数据友好
- 模型效果通常较稳定
- 更容易做特征贡献解释

若以当前工程一致性为优先，建议先采用 LightGBM 作为主方案。

## 6.5 风险等级映射

风险预测模型输出的是概率值，前端展示时需要映射为管理者可读的风险等级。

可先沿用三档分层思路：

- 高风险
- 中风险
- 低风险

阈值细节可在实验中校准，但必须满足：

- 映射规则固定
- 页面、接口、报告统一使用同一标准

## 7. 归因解释设计

## 7.1 目标

归因解释模块用于回答“为什么学生会被判定为高风险或中风险”。

## 7.2 输入

- 风险预测模型
- 单个学生或群体的特征值
- 模型输出概率

## 7.3 输出

- 至少 3 个关键解释维度
- 各维度对风险的影响方向
- 可用于页面和报告展示的解释摘要

## 7.4 建议方法

V1 建议采用：

- SHAP 作为主要解释方法
- 群体平均值对比作为辅助说明
- 规则模板作为解释补充

这种组合的优点是：

- 既有模型层的技术可信度
- 又有业务层的可读性

## 8. 干预建议生成设计

## 8.1 目标

干预建议模块用于将识别结果和解释结果转化为可执行的管理建议。

## 8.2 输入

- 学生个体画像
- 四象限群体标签
- 风险等级与风险概率
- 关键行为因子解释
- 干预规则模板

## 8.3 输出

- 问题概述
- 干预方向
- 建议措施
- 后续观察重点

## 8.4 生成策略

V1 采用分层生成策略：

1. 先由规则模板确定建议骨架
2. 再由 LLM 进行语言润色和个性化表达

采用这一策略的原因：

- 比完全生成式更稳
- 更容易控制输出质量
- 更适合比赛阶段快速落地

## 9. 模型链路与页面链路的对应关系

为了避免模型设计和系统展示脱节，建议统一对应关系如下：

- 特征宽表 -> 个体画像页、群体分析页
- 聚类结果 -> 四象限群体分布、群体分析页
- 风险概率 -> 总览仪表盘、风险预警页、个体画像页
- SHAP 结果 -> 个体画像页、报告页
- 干预建议 -> 个体画像页、报告页

## 10. 评估与验证建议

V1 阶段建议至少保留以下验证内容：

- 风险预测模型 AUC
- 各群体样本占比分布
- 四象限标签映射的可解释性检查
- 关键解释因子是否符合业务常识
- 干预建议是否能与象限类型和关键因子对应

## 11. 当前结论

V1 阶段的模型设计建议收敛为以下一条稳定链路：

多源数据清洗 -> 学期级特征宽表 -> KMeans 聚类 -> 四象限标签映射 -> LightGBM 风险预测 -> SHAP 归因解释 -> 规则模板 + LLM 干预建议

这条链路兼顾了比赛可展示性、技术实现可行性和业务解释能力，能够与前面的范围文档、成果文档和页面文档形成一致闭环。
