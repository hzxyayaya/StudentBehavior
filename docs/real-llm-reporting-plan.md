# 真实 LLM 在线报告链路方案

Last Updated: 2026-04-08

## 1. 目标

将当前主要依赖规则模板生成的学生报告，升级为：

- 规则模板兜底
- LLM 在线增强 / 润色
- 可回退
- 可答辩说明

## 2. 当前状态

当前系统已经具备：

- `top_factors`
- `intervention_advice`
- `intervention_plan`
- `report_text`

但当前核心仍然是模板生成，而不是稳定的在线 LLM 报告链路。

## 3. 总体策略

不推翻现有模板系统，而是在其上增加一层 LLM 增强。

即：

1. 规则模板先生成结构化结果
2. 再把结构化结果交给 LLM 做润色 / 解释增强
3. 如果 LLM 不可用，则回退到模板输出

## 4. 为什么要保留模板兜底

原因：

- 比赛演示需要稳定
- 在线调用可能失败
- API key 可能缺失
- LLM 输出可能漂移

因此，模板不能删，只能作为底座保留。

## 5. LLM 输入设计

不要把原始杂乱数据直接喂给模型。

建议输入应为结构化摘要：

- 学生基本信息
- 风险等级
- 风险概率
- 学业风险 / 行为风险
- 8维结果
- 关键因子
- 趋势变化
- 去向分析结果（未来可选）

## 6. 建议模块

在 `projects/model-stubs` 中新增：

- `llm_reporting.py`

职责：

- 组装 prompt
- 调用 LLM
- 返回报告文本
- 处理失败回退

## 7. Prompt 设计原则

Prompt 应强调：

1. 只能基于输入事实进行总结
2. 不做医学、心理学诊断
3. 建议必须具体、可执行
4. 输出结构稳定

建议输出结构：

- 风险概述
- 核心证据
- 关键解释
- 干预建议
- 后续关注点

## 8. 结果来源标记

建议在报告接口中新增：

- `report_source`

可取值：

- `template`
- `llm`
- `hybrid`

这样前端和答辩时都能解释清楚报告来源。

## 9. demo-api 接入

建议在：

- `/api/students/{student_id}/report`
- `/api/results/intervention-advice`

中增加：

- `report_source`
- `prompt_version`

## 10. 前端展示建议

学生报告页不要只显示一整段文本。

建议继续保留：

- 风险解释
- 关键因子
- 干预建议
- 报告摘要

再补：

- 报告来源标签

例如：

- 模板生成
- 模板 + LLM 润色
- 在线 LLM 生成

## 11. 最小成功标准

这条链完成后，至少要满足：

1. 有真实 LLM 调用入口
2. 有失败回退机制
3. 页面能显示报告来源
4. 没有 LLM 也能稳定演示

## 12. 当前建议

第一版建议做成：

- `hybrid`

即：

- 模板负责结构和事实
- LLM 负责语言增强

而不是完全让 LLM 自由生成。
