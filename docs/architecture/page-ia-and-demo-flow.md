# 页面信息架构与 Demo 主流程

Last Updated: 2026-03-25

## 1. 文档目的

本文档用于定义 V1 阶段系统的页面信息架构与比赛演示主流程，确保后续页面设计、前后端联调和答辩展示围绕同一条叙事主线展开。

本文档重点回答以下问题：

- V1 需要哪些核心页面
- 每个页面承载哪些分析成果
- 页面之间如何形成自然的下钻关系
- 比赛演示时应按什么顺序讲解

## 2. 设计原则

V1 的页面设计不追求“大而全”，而追求以下目标：

- 让评委在短时间内看懂系统主线
- 先展示群体洞察，再下钻个体分析
- 突出“风险预警 + 干预建议”的业务闭环
- 让每个页面都明确服务于一个或多个分析成果

因此，V1 页面数量应控制在最小必要集合内，不额外扩展过多后台管理或边角功能。

## 3. 页面架构总览

V1 建议采用以下 5 个核心页面：

1. 登录页
2. 总览仪表盘
3. 群体分析页
4. 风险预警页
5. 学生个体画像页

如时间允许，可增加 1 个增强页面：

6. 模型训练与结果管理页

其中，前 5 页组成比赛演示的标准闭环；第 6 页属于增强能力，用于展示系统完整性和技术深度。

## 4. 页面与分析成果映射

## 4.1 登录页

页面定位：
系统入口页，用于建立“教育管理分析平台”的第一印象。

主要承载内容：

- 系统名称
- 项目主题摘要
- 角色入口或默认管理者入口

对应分析成果：

- 不直接承载分析成果
- 主要承担系统入口和场景铺垫作用

## 4.2 总览仪表盘

页面定位：
面向学校管理者的第一视角页面，用于展示学校层面的总体风险态势和核心群体结构。

建议模块：

- 总人数与风险分布统计卡片
- 综合测评低等级风险总览
- 四象限群体分布图
- 学院或专业风险对比摘要
- 学期趋势摘要

对应分析成果：

- 成果 2：学生群体画像
- 成果 3：4 类行为模式识别
- 成果 4：综合测评低等级风险预测
- 成果 7：行为演化趋势分析
- 成果 8：学院/专业群体对比与风险分布分析

页面价值：

- 用一页快速建立“系统能看全局”的认知
- 是 Demo 叙事的起点

## 4.3 群体分析页

页面定位：
用于深入展示四象限群体的特征差异、规模分布和管理含义。

建议模块：

- 四象限主视图
- 各象限特征摘要卡
- 各象限风险占比对比
- 典型特征雷达或对比条形图
- 点击群体查看学生明细入口

对应分析成果：

- 成果 2：学生群体画像
- 成果 3：4 类行为模式识别
- 成果 8：学院/专业群体对比与风险分布分析

页面价值：

- 解释“学生群体是怎么被分层的”
- 为后续下钻到风险学生或个体画像建立逻辑桥梁

## 4.4 风险预警页

页面定位：
用于展示高风险和中风险学生的名单、筛选条件和群体分布，体现系统的可操作性。

建议模块：

- 风险等级筛选
- 四象限标签筛选
- 学院/专业筛选
- 风险学生分页表格
- 群体与风险交叉分布摘要

对应分析成果：

- 成果 4：综合测评低等级风险预测
- 成果 8：学院/专业群体对比与风险分布分析

页面价值：

- 把抽象的风险结果转化为可定位、可筛选、可下钻的对象
- 体现系统的管理实用性

## 4.5 学生个体画像页

页面定位：
用于承接从群体到个体的最终下钻，展示单个学生的全景分析结果。

建议模块：

- 基本信息卡片
- 风险概率与四象限标签
- 8 维行为特征雷达图
- 关键行为因子解释图
- 学期趋势变化图
- 个性化干预建议面板

对应分析成果：

- 成果 1：学生个体画像
- 成果 5：关键行为因子归因解释
- 成果 6：个性化干预建议生成
- 成果 7：行为演化趋势分析

页面价值：

- 展示系统如何把群体洞察落到具体学生
- 是“解释 + 干预”能力的核心展示页

## 4.6 模型训练与结果管理页（增强）

页面定位：
展示模型训练、聚类结果和评估指标，作为系统完整性的增强模块。

建议模块：

- 训练参数概览
- 当前模型效果指标
- 各类群体样本数
- AUC、训练样本量、更新时间
- 模型结果摘要

对应分析成果：

- 主要支撑成果 3、成果 4、成果 5 的技术来源说明

页面价值：

- 用于向评委展示系统不是静态页面，而是具有分析链路与模型支撑
- 不应抢占主 Demo 叙事时间

## 5. 推荐的信息架构层级

推荐采用如下层级：

1. 总览仪表盘
2. 群体分析页
3. 风险预警页
4. 学生个体画像页

逻辑关系如下：

- 总览页负责看全局
- 群体页负责解释分层
- 预警页负责定位重点对象
- 个体页负责解释与干预

这是 V1 最自然的一条信息流。

## 6. Demo 主流程

比赛演示建议控制在一条清晰主线内，不在演示中一次讲完所有技术细节。

推荐主流程如下：

### 第一步：从总览仪表盘进入

演示目标：
让评委快速理解系统服务的是“学校管理者的群体洞察”。

建议讲法：

- 学校当前整体风险分布如何
- 哪些学院或专业值得优先关注
- 学期趋势是否出现风险上升
- 学生整体分布在四象限中的情况如何

### 第二步：进入群体分析页

演示目标：
解释四象限模型的业务含义和学生分层逻辑。

建议讲法：

- 为什么采用“投入度 × 稳定性”的四象限结构
- 四类群体分别代表什么样的学生状态
- 哪个群体最需要关注，哪个群体适合作为正向样本

### 第三步：进入风险预警页

演示目标：
展示系统如何把群体洞察转化为具体可管对象。

建议讲法：

- 风险学生可以按风险等级、四象限类型、学院专业进行筛选
- 管理者可以快速定位重点学生或重点群体
- 风险结果不是黑盒，而是可进一步解释的

### 第四步：下钻到学生个体画像页

演示目标：
展示系统如何完成“识别 -> 解释 -> 干预”的闭环。

建议讲法：

- 该学生属于哪个四象限群体
- 为什么会被判定为高风险或中风险
- 哪些关键行为因子推高了风险
- 系统给出的干预建议是什么
- 该学生在最近几个学期中的行为与风险变化如何

### 第五步：补充说明模型训练页（可选）

演示目标：
作为技术补充，展示系统具有真实的模型支撑。

建议讲法：

- 风险模型使用了综合测评低等级标签
- 当前聚类与预测模型有可量化指标
- 系统具备模型训练和结果管理能力

## 7. 推荐的答辩叙事逻辑

为了让演示更加自然，建议按以下句式组织：

1. 学校层面存在哪些总体风险和群体差异
2. 我们如何基于多源数据把学生划分为四类关键群体
3. 我们如何识别高风险学生
4. 我们如何解释风险成因
5. 我们如何输出可执行的干预建议

这比“先讲模型，再讲页面”的技术顺序更适合比赛。

## 8. 页面最小实现建议

若时间有限，建议优先保证以下最小闭环：

- 总览仪表盘可展示风险统计和四象限分布
- 群体分析页可展示四象限定义和群体差异
- 风险预警页可完成筛选与学生列表展示
- 学生个体画像页可展示画像、解释和干预建议

只要这四页打通，V1 的比赛演示主线就成立。

## 9. 当前结论

V1 阶段的页面信息架构建议以“总览 -> 群体 -> 预警 -> 个体”为主路径展开。

这一路径既符合学校管理者的使用逻辑，也最适合比赛答辩中的展示节奏。

后续页面设计、接口设计和前端开发，都应优先围绕这条主链路组织，而不是平均铺开所有功能。
